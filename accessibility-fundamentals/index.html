Accessibility Fundamentals

Understanding the basic principles of web accessibility

Wed, 31 Oct 2018

Accessibility means providing access to all, and when it comes to the web this mainly covers providing a good experience for visually impaired users using screen readers. These read the text from your site aloud, and make use of the html on your page to provide the best experience to users they can.

There are several things you can do as a developer to enhance this experience for them.

Semantic HTML

Semantics: “the branch of linguistics and logic concerned with meaning”

Using semantic html means that the elements you use convey meaning about the content inside them.

Headings

There are 6 heading tags in html - h1, h2, … - and the heirarchy of these matter. An h1 heading is more “important” than an h2 heading. So if your webpage is displaying a news article, the main title might be an h1 and the subtitles in the article h2s. In this case all of the sub-headings using the h2 tag would be considered to be equally “important”.

A screen reader can provide a list of all headings to the user, and use the heirarchy to navigate. They can also allow the user to skip to the previous/next heading.

Do not use a heading just because you prefer its styling for that particular piece of text, you can always change this. The markup you use matters!

Containers

Rather than just using a div for everything, try and use the more semantic elements such as article, nav and aside. See the MDN docs on container elements for the full list.

These provide context and meaning to your page’s content and will make things easier for the screen reader user.

Buttons & Links

Some UI elements come with inbuilt keyboard support, and can be tabbed, clicked and opened through keys alone.

button’s and a links are 2 of these, though you will often see div’s used where a button could have been. A div loses the tabbing functionality (though the tabIndex attribute can be added to enable this), and to support clicking via hitting the enter key you would need to implement a hacky document event listener for keydown. This could all have been much easier if a button element was just used in the first place.

There is a difference between when you should use button and when you should use a. An anchor tag signifies that this will take the user to a new location on the internet, where a button just indicates that if you click this something will happen.

The text in a link or a button should also be able to stand by itself (no “click here”s!). A screen reader can present the user with a list of buttons and links using their inner text, and descriptive text will give them confidence in what they are clicking.

This is better

<a href="/newspapers">Find out more about newspapers.</a>

than this

<p>To find out more about newspapers <a href="/newspapers">click here.</a></p>

Inputs

Make use of labels and tie these to the inputs they relate to. This will provide more information to the user using a screen reader, and also give a greater hit area for your control, so that clicking on a label will de/select its checkbox, which makes things easier.

<label for="name">Enter your name:</label>
<input id="name" type="text">

Images

Use the alt attribute to provide a useful text description of the image. You can use this in conjunction with the title attribute which will provide a more succinct description, and will also be shown to users in a tooltip when they hover over the image.

If the image is not actually important in terms of content, and is just being used for aesthetic reasons, you could use a css background image, set the alt attribute to an empty string (which will signal to screen readers to ignore it) or add the role="presentation" aria attribute (aria roles are covered in more detail below).

Language

Perhaps this doesn’t entirely belong under the “semantic html” category, but the language you use in your website should be clear and easy to read. Don’t overuse slang or jargon, and try to avoid the dash character which is skipped by screen readers.

If you are using acronyms or abbreviations, use the abbr element to provide the expanded version

<abbr title="Papua New Guinea">PNG</abbr>

WAI-ARIA

After you have made your html as semantic as possible, you can further improve the experience for screen reader users by utilising the WAI-ARIA attributes to provide extra semantics.

These are defined in a spec by the W3C and are solely related to accessibility, they will not affect your page in any way other than how it is interpreted by screen readers.

They fall into 3 categories:

  • Role
  • Properties
  • States

Roles

These are used to define an element’s purpose. Some of them directly correlate to html elements, such as role="navigation" for nav, while others do not have a corresponding element at all such as role="tab".

Some of them work in conjunction with others, so an element with the tab role will be inside another with the tablist role, and will be associated with another with the tabpanel role.

<div role="tablist">
  <div role="tab" aria-controls="marsupials">Marsupials</div>
  <div role="tab" aria-controls="rodents">Rodents</div>
</div>
<div id="marsupials" role="tabpanel">
  Here is some info on quokkas and bilbies...
</div>
<div id="rodents" role="tabpanel">
  Here are some cute pictures of squirrels
</div>

See the full list of roles.

Properties & State

These are attributes prefixed with aria- which give extra meaning to your elements. Properties are values that will only be set once, whereas state values could be changed again and again as the user interacts with your application.

They cover things such as aria-required="true" (indicating an input field must be filled in for a form) and aria-disabled="true" (indicating a button or other control is currently disabled).

There is another called aria-live which can be helpful when your application updates some of its content dynamically - this is something that screen readers can struggle with. Use this attribute to signal to a screen reader that an area of content has been updated.

See the full list of state and property values

Conclusion

First, make your HTML as semantic as possible; choose your elements with care and be thoughtful about the text you use. Second, apply aria attributes to fill in any gaps where screen readers may not be getting the full picture.

To check your own application’s accessibility, you can install a11y and run an audit against your site’s URL (works with localhost) or even on your project’s HTML files.
